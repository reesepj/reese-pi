How extensions work in Pi Agent

Extensions are TypeScript modules or Pi packages. They can add tools, slash commands, flags, UI widgets, providers, event hooks, and custom rendering.

Load one for a run

Use -e / --extension for local TypeScript entrypoints.

pi -e ./extensions/coms.ts --name planner
pi -e ./extensions/coms-net.ts --name dev

Auto-load packages

Packages in .pi/settings.json load at startup. This repo pins providers, MCP, memory, todo, Telegram, and Plannotator.

pi install npm:@samfp/pi-memory
pi install npm:pi-mcp-adapter
/reload

Hot reload local work

Put extensions in .pi/extensions/ or ~/.pi/agent/extensions/ for auto-discovery and /reload.

.pi/extensions/my-ext.ts
~/.pi/agent/extensions/my-ext.ts
/reload
Inside Pi: type / to see slash commands. Tools are normally chosen by the model; ask naturally, e.g. “use coms_net_list then ask planner to review this plan.” CLI flags are passed at launch, e.g. --name, --project, --server-url.

Local custom extensions

These are the repo-owned TypeScript entrypoints and helpers you can load with pi -e or through wrappers.

Same-machine coms

extensions/coms.ts lets multiple Pi agents on the same machine discover and message each other through per-project registries and local sockets.

tools/comslocal sockets

Tools: coms_list, coms_send, coms_get, coms_await

Flags: --name, --purpose, --project, --color, --explicit

pi-local-coms --name planner --purpose "Plans work"
pi-local-coms --name dev --purpose "Builds code"

Networked coms-net

extensions/coms-net.ts is the HTTP/SSE successor to local coms. It uses scripts/coms-net-server.ts as a hub and can work across terminals or hosts.

tools/coms-netHTTP/SSE

Tools: coms_net_list, coms_net_send, coms_net_get, coms_net_await

Flags: --name, --purpose, --project, --color, --explicit, --server-url, --auth-token

pi-coms-net-server
pi-coms --name dev
pi-coms-claude --name claude

Verifier loop

apps/verifier/verifiable.ts runs in the builder. It spawns a read-only sibling verifier Pi that inspects claims and can send corrective prompts back.

/verifyverifier_prompttmux

Builder command: /verify. Builder flags: --verifiable, --verifier-agent NAME.

Verifier-side tool: verifier_prompt delivers actionable feedback to the builder via the socket channel.

pi-verifier
# or manually:
pi -e ./apps/verifier/verifiable.ts -e ./apps/verifier/cross-agent.ts --verifiable

Cross-agent command loader

apps/verifier/cross-agent.ts scans .claude/, .gemini/, .codex/, and .pi/agents/ to surface familiar commands and skills inside Pi.

commandsskillsbuilder only

Current project commands: /install, /plan-w-team, /prime. Skills become /skill:name.

pi -e ./apps/verifier/cross-agent.ts
/prime
/plan-w-team "build the feature"

Installed Pi packages

Pinned in .pi/settings.json. These load automatically when Pi starts in this workspace.

pi-claude-cli — Claude Code-backed provider

Routes Pi model calls through the Claude Code CLI. This workspace defaults to defaultProvider: pi-claude-cli, defaultModel: claude-sonnet-4-5, defaultThinkingLevel: high.

pi --provider pi-claude-cli --model claude-sonnet-4-5
/model claude-sonnet-4-5
pi-xai-oauth — Grok provider + xAI tools

Adds the xai-auth provider and Grok models. Also contributes xAI-native tools for search, research, image generation, critique, image analysis, and code analysis.

Common tools: xai_generate_text, xai_web_search, xai_x_search, xai_multi_agent, xai_deep_research, xai_generate_image, xai_analyze_image, xai_critique, xai_code_execution.

/login xai-auth
/model grok-4.3
pi --model grok-4.3 "research this"
pi-mcp-adapter — lazy MCP bridge

Gives Pi access to MCP servers without injecting every MCP tool schema into context. Use the single mcp proxy tool to discover, describe, connect, and call MCP tools.

Commands: /mcp, /mcp setup, /mcp-auth SERVER. Flag: --mcp-config PATH.

mcp({ search: "screenshot" })
mcp({ tool: "chrome_devtools_take_screenshot", args: '{"format":"png"}' })
/mcp setup
@samfp/pi-memory — persistent memory

Learns preferences, project patterns, tool habits, and corrections. Injects relevant memory at session start.

Tools: memory_search, memory_remember, memory_forget, memory_lessons, memory_stats. Command: /memory-consolidate.

memory_search({ query: "commit style" })
memory_remember({ type: "fact", key: "pref.editor", value: "vim" })
@juicesharp/rpiv-todo — live task overlay

Adds a branch-replayed task list the model can manage during multi-step work. Tasks survive /reload and compaction.

Tool: todo. Command: /todos. Statuses: pending, in_progress, completed, deleted.

todo({ action: "create", subject: "Audit wrappers" })
todo({ action: "update", id: 1, status: "in_progress", activeForm: "auditing wrappers" })
@juicesharp/rpiv-ask-user-question — structured questions

Lets the model ask concise, structured clarifying questions instead of guessing. Supports multi-select and side-by-side previews.

Tool: ask_user_question. Use when a concrete decision blocks progress.

ask_user_question({ questions: [{
  question: "Which path should I take?",
  header: "Approach",
  options: [
    { label: "Fast", description: "Smallest safe change" },
    { label: "Deep", description: "Broader refactor" }
  ]
}] })
@llblab/pi-telegram — Telegram bridge

Turns a Telegram DM into a session-local Pi operator console. Supports text, images, files, queues, model controls, voice policy, buttons, and outbound attachments.

Pi commands: /telegram-setup, /telegram-connect, /telegram-disconnect, /telegram-status. Tool: telegram_attach.

Telegram commands: /start, /compact, /next, /continue, /abort, /stop.

/telegram-setup
/telegram-connect
telegram_attach({ paths: ["./artifact.html"] })
@plannotator/pi-extension — plan/code review UI

Provides browser-based planning, annotation, and code-review flows. Also contributes Plannotator skills.

Commands: /plannotator, /plannotator-status, /plannotator-review, /plannotator-annotate, /plannotator-last, /plannotator-archive. Shortcut: Ctrl+Alt+P.

Tool: plannotator_submit_plan while planning mode is active. Flag: --plan.

plannotator_submit_plan({ filePath: "PLAN.md" })

Fast workflows

Copyable launch patterns for the common setups.

Two local agents talk

Open two terminals in this repo.

# Terminal A
pi-local-coms --name planner --purpose "Plans work"

# Terminal B
pi-local-coms --name dev --purpose "Builds code"

In either Pi, ask: Use coms_list, then send planner a request to review this approach.

Networked agent pool

# Terminal A: hub
pi-coms-net-server

# Terminal B/C: clients
pi-coms --name dev
pi-coms-claude --name claude

Use /coms-net --server to inspect hub health and coms_net_list to see peers.

When answering an inbound peer prompt, just answer normally. Do not call coms_net_send to reply.

Proof-first builder loop

pi-verifier

Build normally. The sibling verifier reads the latest session slice after each turn and reports verified, failed, or unsure.

If needed, run /verify to attach manually.

Troubleshooting cues

Wrapper not found

Ensure ~/.local/bin is on PATH and rerun bash scripts/install-workstation.sh.

Tool missing

The current Pi process probably did not load that extension. Launch through the wrapper or add -e ./path.ts, then /reload.

Coms peer missing

Check matching --project, live processes, and --explicit. Use /coms --all or /coms-net --all.
